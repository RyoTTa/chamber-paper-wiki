# Understanding and Optimizing Database Pushdown on Disaggregated Storage

**Venue:** 
**저자:** Hua Zhang, Xiao Li, Yuebin Bai (Beihang Univ.), Ming Liu (UW-Madison)

## 개요

### 1.1 Hardware Trends that Break Pushdown Assumptions

Storage disaggregation에서 computation pushdown은 전통적으로 "데이터 이동 비용 > 연산 비용"이라는 가정에 기반해 왔음. 그러나 최근 10년간의 hardware trend가 이 가정을 완전히 파괴.

**4세대 storage appliance 분석 (Table 1):**

| Generation | Year | CPU | Network | Storage | Net Density (Gbps/core) | I/O Density (IOPS/core) |
|------------|------|-----|---------|---------|------------------------|------------------------|
| Gen1-SA | 2010 | 2×Xeon E5504 (4C), 64GB DDR3 | 8Gb FC | SATA HDD | 1.0 | 37.5K |
| Gen2-SA | 2014 | 2×Xeon E5-2640v2 (8C), 96GB DDR3 | 25GbE | SATA SSD | 1.6 | 75K |
| Gen3-SA | 2018 | 2×Xeon Gold 6130 (16C), 192GB DDR4 | 100GbE RoCEv2 | NVMe Gen3 | 3.2 | 125K |
| Gen4-SA | 2022 | 4×ARM A72 (8C), 64GB DDR4 | 400GbE RoCEv2 | NVMe Gen3 | 12.5 | 500K |

Network density와 I/O density가 **order of magnitude 증가** (1→12.5 Gbps/core, 37.5K→500K IOPS/core), 반면 per-core CPU frequency/memory BW는 정체.

### 1.2 Empirical Evidence: Pushdown Failures on Modern Nodes

Gen3-SA/Gen4-SA에서 SSB + TPCH benchmark 수행 결과:
- **Heuristic-based:** Gen3-SA 40% degradation, Gen4-SA 55% degradation
- **Cost-driven:** Gen3-SA 36% degradation, Gen4-SA 45% degradation

Query runtime breakdown (Figure 2): Gen3-SA/Gen4-SA에서 **Compute가 76.8%/74.1%** 차지 (Gen1-SA/Gen2-SA에서는 Network+I/O가 65–75% 차지). **Bottleneck shift: network/I/O → compute.**

## 방법론

**Two design principles:**
- **P1 (Lazy Evaluation):** Pushdown decision을 operator scheduling 직전까지 지연 → runtime 정보 기반 의사결정
- **P2 (Trading network/I/O for compute):** 과잉 network/I/O BW를 scarce compute 자원과 교환

### 3.1 Table-aware Learning-based Cost Estimator (TLCE, §4.2)

**Three-step estimation:**

1. **Equidistant Sampling:** `k = size(input_op) / size(sample_op)` → select first row + every k-th rows. Sampling ratio 10% (marginal accuracy gain threshold).
2. **Eager Input Estimation:** First received data message size × scaling factor. Multiple upstream producers → per-type scaling (`⟨m·|A_i|, n·|H_i|⟩`).
3. **Pre-Scan Balancing Operator (skew mitigation):** Partition → `m = partitionSize/segmentSize` uniform segments → `scanner_i` reads `segment_{i mod n}` via `seekg(offset) + read(segmentSize)`. Fine-grained segmentation ensures near-uniform distribution.

**Regression model:** `T_op = k_1·T_sample_op + k_2·size(input_op)`.
- Parameter vector `(k_1, k_2)` obtained offline via random query generator + incremental learning (River)
- **Accuracy:** Max/min/avg RMAE = 0.27/0.07/0.16 across all SSB queries

**Skew handling results:** Segment size 9000 bytes → accuracy 98.6%(skew=0), 98.1%(skew=2), 94.9%(skew=4). Improvement: 19.6%/62.7%/73.4% over conventional scan.

### 3.2 Admission Control (AC, §4.3, Algorithm 1)

**Core idea:** Historical prediction error를 이용해 storage node의 overload 상태를 추론.

- Adaptive parameter `A`: `T_est = A · T_p`
- Loss: `L = (T_target − A·T_p)²`
- Gradient: `∂L/∂A = −2·(T_target − A·T_p)·T_p`
- Online update: `A ← A − η·gradient` (η = 0.1 optimal)

**Effect:** Larger A → heavier load signal → pushdown threshold stricter → fewer concurrent operators → stay within interference-free window.

**Accuracy improvement:** RMAE max/min/avg: 0.21/0.04/0.19 (vs. 0.38/0.06/0.27 without AC).

### 3.3 DRAM-SSD Hybrid Ballooned Table (HBTable, §4.4)

**Motivation:** Gen3-SA/Gen4-SA DRAM 제한 (64GB DDR4). System-level memory throttling would hurt performance.

**HBTable data structure (Figure 6-a):**
| Component | Content |
|-----------|---------|
| Manifest | Operator ID, Priority, Tuple Metadata (column name, data type), SSD Head/Tail LBA, DRAM Usage |
| DRAM region | Resizable circular buffer, entries proportional to operator demand |
| SSD region | Resizable circular buffer, LBA-addressed |

**Operations:**
- **append:** Check memory pool → if space: append to DRAM tail. Otherwise: ballooning (memory lending) → if still insufficient: append to SSD tail
- **pop:** Check DRAM Usage → return head if pending; else fetch from SSD region
- **lend (ballooning):** High-priority table borrows from low-priority when below threshold. Lender spills to SSD.
- **redistribute:** Return borrowed memory → redistribute to all active HBTables proportional to priority

### 3.4 Critical Path-Driven Operator Scheduler (CPSched, §4.5, Algorithm 2)

**Problem formalization:** DAG G = (O, E) of operators. Objective: minimize makespan (max completion time). NP-hard (Job Shop Scheduling).

**Path classification:**
- **L:** Longest execution (critical) path → highest priority
- **U:** Urgent bypass → determines readiness of L-operators (e.g., JoinBuild before JoinProbe)
- **C:** Common bypass → lowest priority

**Algorithm (CPSched):**
```
If ResourceDemand(op) > R_total:
  while op_low = activeQueue.head():
    if pri_low > pri: return (higher-priority already running)
    else: push op_low to readyQueue, release resources
activeQueue.append(op), readyQueue.pop(op)
```

Within same category: topological order. Blocked operators released and re-appended to readyQueue.

## 핵심 기여

**Core contributions:**
1. **Systematic root cause analysis:** Network/I/O density growth → bottleneck shift to compute + 3 root causes (table structure, interference, scheduling) quantified across 4 hardware generations
2. **TLCE:** Table-aware online cost estimator via equidistant sampling + incremental learning → RMAE 0.16 avg
3. **Admission Control:** Gradient-descent-based adaptive parameter → dynamically confines operators within interference-free window
4. **HBTable:** Memory ballooning with DRAM-SSD hybrid circular buffers → implicit memory capacity expansion
5. **CPSched:** Critical-path-aware operator scheduling with DAG dependency classification (L/U/C)

**Broader significance:** "Pushdown = always beneficial" 가정이 더 이상 유효하지 않음을 정량적으로 입증. Disaggregated storage design에서 policy-driven approach에서 compute-aware execution design으로의 paradigm shift를 제시. Network/I/O bandwidth를 abundant resource로, compute를 scarce resource로 재정의하는 새로운 cost model 확립.

## 주요 결과

- Base: FPDB (open-source pushdown DB) + extended with TapDB modifications
- Learning-based cost estimator: River library
- Runtime: CAF (C++ Actor Framework) with operator parallelization + CPU allocation
- Cache disabled, pushdown logic retained and integrated
- S3 replaced with custom object store
