# Cxlalloc: Safe and Efficient Memory Allocation for a CXL Pod

**Venue:** ASPLOS '26 (Vol. 2), March 22–26, 2026, Pittsburgh, PA, USA
**저자:** Newton Ni, Yan Sun, Zhiting Zhu, Emmett Witchel

## 개요

CXL pod는 8–16개의 host가 하나의 CXL-attached memory device를 cacheline granularity로 공유하는 구조. 이런 환경의 memory allocator는 기존 allocator가 해결하지 못한 **세 가지 근본적 문제**를 동시에 해결해야 함:

### Challenge 1: Limited Inter-Host HWcc

CXL 3.0+ spec은 inter-host HWcc를 정의하지만, 구현 비용과 복잡성으로 인해 실제 광범위한 지원 여부는 불투명. 현재 두 개의 hardware prototype 모두 HWcc 미지원. 관련 연구는 세 가지 HWcc 모델을 가정:
- **Full HWcc**: 모든 영역에서 CAS 등 atomic operation 가능
- **Limited HWcc (Figure 1A)**: 작은 contiguous region만 HWcc 유지
- **No HWcc (Figure 1B)**: Near-memory processing(NMP) logic으로 custom mCAS 구현 필요

기존 allocator들은 thread-local metadata와 global metadata를 intersperse하여 배치 → limited HWcc에서는 HWcc 영역을 낭비하거나, 분리 시 correlation 어려움.

### Challenge 2: Cross-Process Sharing

여러 host의 서로 다른 process가 하나의 shared CXL memory heap을 동시에 접근. 두 가지 pointer consistency(PC) 문제:

- **PC-S (Spatial):** 서로 다른 process에서 pointer가 같은 physical memory를 가리켜야 함. Concurrent `mmap`이 서로 다른 virtual address를 반환할 수 있으며, overlapping 가능.
- **PC-T (Temporal):** 한 process에서 allocate된 memory가 즉시 다른 process에서 dereference 가능해야 함. 새 memory mapping은 다른 process에 자동으로 visible하지 않음.

기존 접근의 한계: Fixed heap size [1, 72], maximum allocation size 제한 [68], mmap 불가.

### Challenge 3: Partial Failure

Host 수 증가에 따라 partial failure 확률 증가. 기존 PM allocator들의 접근은 한계:
- **Garbage collection [14, 16]**: Recovery 중 모든 thread가 heap 접근 불가 (blocking)
- **Locks [23, 52]**: Crash of lock holder → live threads blocking
- **Total failure model**: Quiescent period 가정, CXL partial failure와 불일치
- **Reference counting [68]**: Read-heavy workload에서도 contention 유발, HWcc metadata가 allocation마다 embed됨

## 방법론

### 1. Core Architecture (§3.1)

**Three heaps (Figure 2):**

| Heap | Size Range | Backing |
|------|------------|---------|
| Small | 8B – 1KiB | Slab allocation |
| Large | 1KiB – 512KiB | Slab allocation (same design) |
| Huge | 512KiB+ | Individual memory mappings |

**Slab allocation design (Figure 3):**
- Data region을 fixed-size slab으로 분할 (small: 32KiB, large: 512KiB)
- Slab은 free list에 연결 (intrusive linked list)
- Three free list types: global (SmallGlobal.free), thread-local unsized, thread-local sized
- 각 slab metadata는 두 descriptor로 분리: **SWccDesc** (owner, class, free bitset, next) + **HWccDesc** (remote counter)

**Allocation path:**
1. Thread-local sized free list 확인 → 비어있으면 순서대로: thread-local unsized → global → heap extension (SmallGlobal.len 증가)
2. Slab 획득 후 초기화: owner=자신, class=요청 size class, free=full bitset, remote=total blocks
3. SWccDesc.free에서 block 할당. Slab이 full이면 sized free list에서 unlink.

**Deallocation path:**
- **Owner = deallocating thread**: Local free path → SWccDesc.free update in place. Full→non-full이면 sized free list로 복귀. Empty면 unsized로 이동.
- **Owner ≠ deallocating thread**: Remote free path → CAS로 HWccDesc.remote decrement.

**Slab state transitions (Figure 4):**
- **Active** → (full + no remote free) → **Detached** (owner 유지, unlinked)
- **Active** → (full + remote free 있음) → **Disowned** (owner 제거, unlinked)
- **Detached/Disowned** → (remote counter=0) → remote freed slab을 다른 thread가 steal

### 2. Limited HWcc 지원 (§3.2)

**Metadata partitioning:**
- Small heap: HWcc = SmallGlobal(8B) + HWccDesc(2B/slab → 8B aligned with detectable CAS). Small slab 32KiB당 8B = 0.024%.
- Huge heap: HWcc = HugeGlobal(8KiB constant).
- 전체 HWcc memory: 평균 total memory 대비 **0.02%**, ralloc의 HWcc usage 대비 **7.1%**.

**Remote free protocol:**
- Counter 기반 설계로 HWcc memory 최소화. Bitset/free list 대신 counter 사용 → 어떤 block이 freed되었는지 정보 손실 → **entire slab이 remotely freed될 때까지 block 재사용 불가**
- **Disowned state**: slab에 remote free가 하나라도 있으면 disowned로 전환 → 이후 모든 free가 remote path를 탐 → whole slab reclaim 가능
- 병리적 경우: 많은 slab에서 적은 수의 local free + remote free, thread가 해당 size class 사용 중단 → fragmentation. 일반적이지 않으며 평가에서 과도한 fragmentation 미발견.

**SWcc protocol (§3.2.2):**
- Thread는 CPU cache에 SWcc metadata 유지 가능 (SWcc memory는 HWcc 미지원이지만 CPU caching은 허용)
- Manual flush/fence 타이밍:
  - **Writer**: Ownership change 전에 flush+fence (global free list로 이동, detached/disowned 전환 시)
  - **Reader**: Global free list pop → load 전 flush+fence. Deallocation 시 owner load → flush/fence 불필요 (case analysis로 safety 증명):
    1. Owner = self (cache=memory): safe
    2. Owner = self (cache≠memory): safe
    3. Owner = self in cache, not in memory: impossible (ownership give-up 시 반드시 flush했을 것)
    4. Owner ≠ self: remote free path의 decrement는 stale owner와 무관하게 정확 (counter가 down-count이므로)

### 3. Cross-Process Sharing (§3.3)

**Virtual address space reservation:**
- 초기화 시 각 process에서 `mmap(PROT_NONE)`으로 large contiguous virtual address region 예약 → offset pointer를 위한 PC-S 보장
- Gray regions in Figure 2.

**Signal handler for PC-T:**
- SIGSEGV handler 설치 → unmapped pointer dereference 시 heap metadata 검사 → valid allocation이면 memory mapping 설치 후 재실행
- Synchronous barrier design을 기각한 이유: (a) subset of processes만 접근하는 mapping에도 global overhead, (b) concurrent mapping update 불가, (c) partial failure에서 live thread blocking.

**Small heap extension (§3.3.1):**
- SmallGlobal.len을 atomically 증가 → HWccDesc, SWccDesc, data 영역에 대해 3개의 memory mapping 생성
- Monotonic extension (never unmap). MADV_REMOVE로 physical memory 반환.

**Huge heap + Hazard offsets (§3.3.2):**
- Reservation array가 coarse-grained virtual address region ownership 관리
- 각 thread는 owned region을 interval tree로 추적
- **Hazard offset protocol** (hazard pointers [51] 변형):
  1. Memory mapping 전에 hazard offset publish
  2. Memory mapping unmap 후 hazard offset 제거
  3. HugeDesc.free + offset이 어떤 hazard list에도 없으면 reclaim 가능
- Validation step 불필요: use-after-free는 correct program에서 발생하지 않음.

## 핵심 기여

Cxlalloc은 **CXL pod를 위한 최초의 포괄적 memory allocator**. 세 가지 핵심 contribution:
1. **Limited HWcc 지원**: Metadata를 HWcc/SWcc로 분리, SWcc protocol로 manual coherence → HWcc memory를 total 대비 0.02%로 최소화
2. **Cross-process pointer consistency**: Offset pointers + virtual address reservation + SIGSEGV signal handler + hazard offsets로 PC-S/PC-T 동시 보장, huge allocation 지원 (타 allocator 불가)
3. **Non-blocking partial failure tolerance**: Lock-free data structures + detectable CAS + atomic state redo로 live thread blocking 없이 recovery

Mimalloc 대비 93.9% throughput 유지, mCAS로 HWcc 없는 CXL device에서 80% throughput 달성. 오픈소스 (https://github.com/nwtnni/cxlalloc). Tigon [33], CXLMC 등 후속 연구의 기반 allocator로 채택됨.

## 주요 결과

**Non-blocking crashes:**
- Lock-free data structures: 4개만 multiple-writer (heap length, global free list, HwccDesc, reservation array) → 모두 single CAS operation
- Single-writer data structures (thread-local free lists): crash 시 inconsistent → recovered thread가 복구, 다른 thread에 invisible

**Non-blocking recovery:**
- 각 operation 진입 시 8 bytes의 atomic in-place state update → idempotent redo 가능
- Small heap 예: slab transfer 시 operation ID(4b) + slab index(32b) + size class(8b) 기록
- **Detectable CAS [10]**: Lock-free data structure operation에 사용. CAS target에 thread ID(16b) + version(16b) embedding → recovery 시 global help array로 성공 여부 확인
- Huge heap: Deterministic reconstruction 가능 (reservation array + descriptor list)

**PM allocator와 비교 (Figure 7):**
- Memento recoverable data structure에 1M objects insert/remove, 0/1/2 thread crashes
- Ralloc: blocking GC 또는 memory leak 선택
- Cxlalloc: non-blocking recovery, no leak

## 구현 (§4)

**mCAS Prototype (Intel Agilex 7 FPGA):**
- Device-biased memory region을 NMP가 관리 (uncachable)
- Two special regions: spwr (64B write: expected+swap+target) + sprd (16B read: result)
- NMP는 sprd 수신 시 mCAS 실행, concurrent conflicting operation 감지 → serialize
- 16 threads 기준: sw_flush_cas 대비 p50 latency 17.4%, p99 20% 낮음
- 1 thread: hw_cas p50 = 2.3μs (sw_flush_cas보다 느림, ASIC version에서 개선 기대)

**Heap initialization:**
- Zeroed memory = valid initialized heap → external coordination 불필요 (부트스트래핑 문제 해결)

**Detectable CAS overhead:** remote free metadata 2B→6B (8B aligned) per slab

## 평가

### 방법론

| 항목 | 내용 |
|------|------|
| **Machine 1 (Chameleon)** | 2× Intel Xeon Platinum 8380, 80 cores, 128 GiB DDR4, NUMA node 0 only |
| **Machine 2 (CXL HW)** | Intel Xeon 8568 (48 cores, Emerald Rapids), Intel Agilex 7 CXL Type-2, PCIe 5.0 x16 |
| **CXL latency** | Read: 357ns (local: 112ns), BW: 19.9 GB/s (local: 114 GB/s) |
| **Workloads** | YCSB (Load/A/D), memcached traces (MC-12/15/31/37), thread-test, xmalloc |
| **Baselines** | mimalloc, boost, lightning, cxl-shm, ralloc (Table 1) |
| **Config** | 10 processes, 1-8 threads/process, 64 GiB shared memory file |
| **Metrics** | Throughput, PSS (proportional set size), HWcc usage ratio |

### YCSB / Memcached Traces (Figure 8)

- Boost, lightning: global mutex로 unscalable
- Cxl-shm: skewed workload에서 reference counting contention, small allocation에서 24B inline header overhead
- **Mimalloc / ralloc / cxlalloc**가 top tier:
  - Cxlalloc: 평균 mimalloc throughput의 **93.9%**, ralloc은 90.9%
  - Cxlalloc HWcc memory: avg **0.02%** of total, ralloc 대비 **7.1%**

### Microbenchmarks (Figure 9)

- **thread-test:** Cxlalloc = mimalloc의 47% (mimalloc의 intrusive linked list fast path에 못 미침). Ralloc = 41%.
- **xmalloc (producer-consumer):** Cxlalloc = mimalloc의 81%, ralloc = 106% (ralloc은 high thread count에서 partially full slab을 global free list로 반환하여 contention 발생)
- HWcc memory: thread-test에서 2.5%, xmalloc에서 0.09% (ralloc 대비 9.4%, 9.5%)

### Huge Allocations (Figure 10)

- **Novel feature**, 모든 baseline은 crash 또는 30분 내 미완료
- Thread-test: thread count 증가에 flat (OS memory mapping work가 bottleneck)
- Xmalloc: hazard offset operation exercise, 안정적 성능

### CXL Hardware (§5.4, Figure 12)

- **threadtest-small:** cxlalloc-hwcc (CXL+HWcc)는 local DRAM과 유사. **Cxlalloc-mcas = cxlalloc-hwcc의 80%**, ralloc-mcas 대비 **10-99×** (SWcc protocol이 metadata를 cache에 유지 가능)
- **xmalloc-small:** Cxlalloc-mcas = cxlalloc-hwcc의 1% (모든 remote free가 mCAS). Below 8 threads: ralloc-mcas 우위 (partial slab sharing). 24 threads: cxlalloc-mcas가 ralloc-mcas 대비 **9.9×** (slab sharing이 mCAS contention 증가)

### Partial Failure Overhead

- Cxlalloc vs cxlalloc-nonrecoverable: throughput 차이 **0.3%** only (recovery logic overhead minimal in fast path)
- thread-test에서 non-recoverable 대비 94.7%
- xmalloc에서 88.4% (detectable CAS cost)
