# STRAW: Stress-Aware WL-Based Read Disturbance Management for High-Density NAND Flash Memory

**Venue:** 
**저자:** Myoungjun Chun, Jaeyong Lee, Inhyuk Choi (SNU), Jisung Park (POSTECH), Myungsuk Kim (KNU), Jihong Kim (SNU)

## 개요

### 1.1 Read Disturbance in High-Density NAND Flash

3D NAND density 증가 (321 WL layers, TLC/QLC)는 read disturbance 문제를 심각하게 악화.

**Mechanism (Figure 2):** Read operation 시 target WL에는 `V_ref`를, non-target WL에는 `V_pass`(>6V)를 인가 → non-target WL cells가 slightly program됨 → 누적 시 `V_th` shift → bit errors.

**Density impact:**
- 321-layer 3D TLC: block size 120 MB (vs. 2D MLC: 5 MB) → 24× more data disturbed per read
- QLC read latency 90μs (vs. SLC 20μs) → FN tunneling: disturbance ∝ `V_pass` × latency
- Longer string → higher required `V_pass`

### 1.2 현장 분석: RR이 SSD 수명과 성능에 미치는 실제 영향 (Figure 3)

3종 off-the-shelf SSD에 4KB random read (QD=32, 12시간) Fio 실험:

| SSD Type | Internal writes (after 30TB host reads) | Read latency |
|----------|----------------------------------------|--------------|
| 2D MLC | Minimal | Stable |
| 72-layer 3D TLC | Moderate | Spikes up to 10ms |
| 176-layer 3D QLC | **8.7 TB** | Spikes up to 10ms |

**수명 계산:** 200 TBW QLC SSD, 50 MB/s read-only workload → **144일 소진** (typ. 3-year warranty 대비 극히 짧음).

### 1.3 기존 RR 접근법의 두 가지 근본적 한계

**Limitation 1: Block-level Granularity (Figure 4, 5)**

| Property | Detail |
|----------|--------|
| Adjacent WL `V_pass` | `V_passH` ≈ 10% higher than `V_pass` for non-adjacent |
| Per-WL tolerance variance | Worst WL: 403K reads, Best WL: 559K reads (within same block) |
| Pattern PA (hot single WL) | Data loss at **54,560 reads** (worst WL = WL36) |
| Pattern PB (uniform sequential) | Data loss at **518,420 reads** |
| Conservative `RC_MAX` | Must be set to 54,560 (worst-case) → 9.5× premature RR for PB |

**Limitation 2: Reactive Manner (Figure 6)**

160개 real NAND flash chip (5 chip types: 2D MLC/TLC, 3D MLC/TLC/QLC) characterization:

| Chip type | Max tolerable reads @1K PEC (PA pattern) |
|-----------|------------------------------------------|
| 2D MLC | 457K |
| 2D TLC | 258K |
| 3D MLC | 195K |
| 3D TLC | 121K |
| 3D QLC | **54K** (88.2% lower than 2D MLC, 79% lower than 2D TLC) |

→ High-density NAND에서 ideal RR조차도 massive data reclamation 필요.

## 방법론

### 3.1 Three Key Data Structures

| Structure | Content |
|-----------|---------|
| **RPT** (Read-reclaim Parameter Table) | `ERC_MAX`, `α`, `β_2.5`, `β_5` for each `⟨PEC, WL_group⟩` pair (offline profiling) |
| **REC** (Resource-Efficient Counters) | Per-block counters: per-WL RC + block-level RC (Space-Saving algorithm) |
| **PVT** (Pass-through Voltage Table) | Minimum-safe `V_pass` per `⟨PEC, adjacent-WL validity, R_INV⟩` |

### 3.2 Space-Saving (SS) Algorithm for REC

Naive per-WL counter: **125 MB** for 2TB SSD. SS solution:
- `N` counters per block (default: 32 for 704-WL block)
- On read: increment if entry exists; else replace `min_count` entry with current WL index
- Query: return counter value if entry exists; else return `min_count`
- **Guarantee:** never underestimates (no data corruption risk, only premature RR possibility)

| REC size (per 704-WL block) | Space reduction | Extra RR page copies |
|----------------------------|----------------|----------------------|
| 32 entries | **95.5%** | +74.4% |
| 64 entries | **91%** | +26.6% |
| Unconstrained | 0% | 0% |

### 3.3 Operational Flow

**Per-read (Figure 15):**
1. Query PVT → select minimum-safe `V_pass` mode for non-adjacent WLs
2. Issue `SET FEATURES` command (158 vendor-specific feature addresses, 4×8-bit params each → 642 bits for 321-WL chip)
3. Update REC: `RC[BLK_tgt] += β_i` (weighted by applied `V_pass`), `RC[WL_tgt] += 1`

**Periodic check (per 1K block reads):**
For every valid WL_i in BLK_tgt:
1. Look up `RC[WL_{i-1}]`, `RC[WL_i]`, `RC[WL_{i+1}]`, `RC[BLK_k]` from REC (Figure 16)
2. Compute: `RC_adj = RC[WL_{i-1}] + RC[WL_{i+1}]`, `RC_non-adj = RC[BLK_k] − RC[WL_i] − RC_adj`
3. `ERC[WL_i] = α × RC_adj + RC_non-adj`
4. If `ERC[WL_i] + α·interval > ERC_MAX` → WL_i is heavily disturbed → reclaim

**Example:** WL_i in Best group at 2K PEC:
- `RC_adj = 67K`, `RC_non-adj = 872K` → `ERC = 7.7×67K + 872K = 1401K`
- `ERC + interval = 1401K + 8K = 1409K < 1806K → not disturbed`

## 주요 결과

### 4.1 Methodology

| Item | Details |
|------|---------|
| **Emulator** | NVMeVirt extended with RR triggering based on real chip data + `N_ERR`-dependent read latency |
| **SSD config** | 480GB; 8ch×2dies×4planes; 176-layer×4WL/layer; 232 blocks/plane; 2112 pages/block |
| **Latencies** | `tREAD`=40μs, `tPROG`=380μs, `tERASE`=3500μs, `tSF`=52ns |
| **Interface** | 8.0 GB/s external (PCIe 4.0 4-lane), 2.0 GB/s channel |
| **Workloads** | FIO-1 (98% rand read), FIO-2 (71% rand read), YCSB-A (70% mixed), YCSB-E (97% rand), Filebench (98% seq), Lumos (91% rand), Llama (99% seq) |
| **PEC levels** | 0K, 1K, 2K |
| **Configs** | Baseline, Cocktail, Pagetype, STRAW⁻ (WR² only), STRAW (both), STRAW+Cocktail |

### 4.2 RR Efficiency (Figure 17)

| PEC | STRAW vs. Baseline | STRAW⁻ vs. Baseline |
|-----|--------------------|--------------------|
| 0K | **−90.0%** RR page copies | — |
| 1K | **−92.2%** | — |
| 2K | **−93.6%** | — |

**Cocktail / Pagetype baseline comparisons:**
- Pagetype: page-type-level RR (e.g., LSB 1.3× più vulnerable) → threshold must still be conservative → limited gain
- Cocktail: redistributes hot pages → effective only on temporal-locality workloads (e.g., YCSB-A)
- **STRAW+Cocktail:** up to **−31%** additional reduction over STRAW alone (25.8% avg on read-dominant workloads)

### 4.3 Read Tail Latency (Figure 18)

| PEC | STRAW 99.9th %-ile reduction vs. Baseline | STRAW⁻ reduction |
|-----|-------------------------------------------|------------------|
| 0K | **−65.2%** | −56.1% |
| 1K | **−71.1%** | −64.0% |
| 2K | **−75.6%** | −69.4% |

Particularly effective on YCSB-E, Llama (high read intensity → RR無法 defer).

### 4.4 Overhead Analysis (Figure 20)

| Metric | Finding |
|--------|---------|
| **Lifetime (block erasures)** | STRAW reduces total erasures **53.6% avg** across mixed workloads (FIO-2, YCSB-A) |
| **Read latency (precharge)** | `V_pass`=7.92V/7.15V worst-case: ≤**4%** precharge latency increase; total read latency impact ≤**1.2%** |
| **Read energy** | Worst-case (all WLs invalid except target): +2.5% read energy. Typical: smaller due to `V_pass` reduction on valid WLs |

### 4.5 Feasibility

| Aspect | Rationale |
|--------|-----------|
| WL switching | Chips already apply `V_passH` to adjacent WLs regardless of location |
| Multiple `V_pass` sources | Existing for WL precharge during reads → per-WL distinct voltage |
| Programmability | `V_pass` levels tunable via post-fabrication fuse programming (already used in Samsung, Seagate) |
| `SET FEATURES` | 158 vendor-specific feature addresses → 21 addresses encode 4 modes/WL for 321-WL chip |
