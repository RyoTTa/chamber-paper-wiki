# ZipServ: Fast and Memory-Efficient LLM Inference with Hardware-Aware Lossless Compression

**Venue:** 
**저자:** Ruibo Fan, Xiangrui Yu, Xinglin Pan, Zeyu Li, Weile Luo, Qiang Wang, Wei Wang, Xiaowen Chu (HKUST-GZ / HKUST / HIT Shenzhen)

## 개요

LLM serving에서 **GPU memory capacity와 bandwidth가 가장 큰 병목** (주로 weight read traffic). Lossless compression은 bit-exact reproduction을 보장하면서 memory footprint를 줄일 수 있으나, 기존 방법은 모두 GPU architecture mismatch로 인해 오히려 **inference slowdown**을 유발.

**구체적 데이터:**
- Figure 1: L40S GPU GateUp_proj layer에서 decoupled decompression만으로 core inference time의 **1.56–3.44× overhead** 발생
- BFloat16 weight exponent 분포 분석 결과 (Figure 2): top-3 most frequent exponent가 전체 weight의 **67% 이상** 차지; top-7 exponent가 **95% 이상** (LLaMA-3 96.4%, Mistral-24B 97.4%); exponent field entropy 2.57–2.74 bits (이론적 CR ≈ 1.51×)
- 3,875개 weight matrix 분석: **99.6%** 에서 top-7 exponent가 numerically contiguous (key structural property)

**Two-level mismatch:**
1. **Kernel-level:** Entropy codec (Huffman, ANS) → variable-length bitstream → serialized data-dependent decoding → **SIMT divergence**, compute underutilization. L40S에서 DietGPU 43.7%, DFloat11 76.5%만 peak mem BW 달성
2. **System-level:** Decoupled pipeline → decompress to global memory first → **redundant memory traffic**. Roofline 분석 결과 (Figure 5): CI degradation **~62%** vs. uncompressed GEMM (M=K=4096, BS 8–64)

## 방법론

| Component | LOC | Language |
|-----------|-----|----------|
| TCA-TBE compressor + ZipGEMM kernel | ~2.5K | CUDA/C++ |
| vLLM integration (loader + linear modules) | ~1.0K | Python (+ PyBind11) |
| **Total** | **~3.5K** | |

- Compilation: GCC 11.3, NVCC 12.4 (NVCC 12.8 for RTX5090)
- Integration: Extended vLLM model loader + linear execution modules → .so library via C++ API

**Offline compression cost:** LLaMA-3.1-8B compression ≈ **2.5 min** on 16-core Xeon 8352V (one-time operation).

## 주요 결과

### 4.1 Methodology Overview

| Item | Details |
|------|---------|
| **Platforms** | ① 4×RTX4090 (24GB, Ada, CC 8.9) + Xeon 8352V; ② 4×L40S (48GB, Ada) + Xeon 6230R; ③ RTX5090 (32GB, Blackwell, CC 12.0) |
| **Models** | LLaMA-3.1 (8B, 70B, 405B), Qwen2.5 (7B, 14B, 32B, 72B), Gemma3 (12B, 27B), Mistral (24B, 123B) |
| **Baselines** | ① cuBLAS_TC v12.4.5, ② DietGPU (rANS), ③ nvCOMP (rANS), ④ DFloat11 (Huffman) |
| **Kernel workloads** | QKV_proj, O_proj, GateUp_proj, Down_proj, LM head; BS 8/16/32; 100 warmup + 1000 timed |
| **E2E benchmarks** | vLLM, Transformers, DFloat11; BS 8/32; output lengths 128–2048 tokens; 10 runs each |

### 4.2 Kernel-level Performance

| Platform | ZipGEMM vs. cuBLAS_TC | DietGPU | nvCOMP | DFloat11 |
|----------|----------------------|---------|--------|----------|
| RTX4090 | **1.31× avg** (max 1.71×) | 0.17× | 0.19× | 0.28× |
| L40S | **1.36× avg** (max 2.21×) | 0.20× | 0.23× | 0.34× |
| RTX5090 | 1.34×–1.87× | — | — | — |

**Layer-wise (LLaMA3.1 on L40S):** GateUp_proj 1.39×, Down_proj 1.64×. Small layers (O_proj of 8B) can slow to 0.79× (minor FLOPs). Block-level: 1.35× (8B), 1.48× (405B).

### 4.3 Micro-analysis (RTX4090, Nsight Compute)

| Metric | ZipGEMM | Impact |
|--------|---------|--------|
| DRAM reads | **−29.3%** | TCA-TBE format validation |
| ALU utilization | 66.0% | Price of on-the-fly decoding |
| Tensor Core utilization | **71.6% of cuBLAS** | Compute preserved |
| Shared memory bank conflicts | ~4.7K | vs. millions in DietGPU |

### 4.4 Standalone Decompression (ZipServ-Decomp)

LLaMA3.1-8B / Mistral-24B Transformer block full decompression:
- vs. DietGPU: **2.14×**
- vs. nvCOMP: **1.83×**
- vs. DFloat11: **1.10×**

### 4.5 End-to-End Inference

| Model | Platform | vs. vLLM latency | vs. vLLM throughput | Weight reduction |
|-------|----------|------------------|---------------------|------------------|
| LLaMA3.1-8B | 1×RTX4090 | **−17.60%** avg | **1.22×** | 14.96→10.83GB (72.4%) |
| Mistral-24B | 2×L40S | included in avg | — | 43.92→31.30GB (71.3%) |
| LLaMA3.1-70B | 4×L40S | included in avg | — | 131.56→93.52GB (71.1%) |

**Best case:** LLaMA3.1-8B, BS 32, 2048 output tokens → **1105 tok/s, 1.66× over vLLM**.

### 4.6 Breakdown Analysis (LLaMA3.1-8B, RTX4090, SeqLen 1024)

| Component | vLLM | ZipServ |
|-----------|------|---------|
| Linear layers (GEMM) | 24.99 ms (83.6%) | **14.76 ms** (1.69×) |
| Attention | 3.02 ms | 3.02 ms |
| Other | 1.88 ms | 1.88 ms |
| KV cache | 5.07 GB | **8.60 GB** (1.70×) |

Saved 3.78 GB weights → allocated to KV cache → throughput gain.

### 4.7 Cross-Generation Performance (Figure 14, Table in paper)

| Scenario | Standard | ZipServ |
|----------|----------|---------|
| RTX4090 vs. A100 (LLaMA3.1-8B) | — | RTX4090 ZipServ 0.195ms > A100 cuBLAS 0.215ms (9.3% faster) |
| RTX4090 vs. A100 (Mistral-24B) | — | ZipServ 0.530ms vs. A100 0.516ms (2.7% slower) |
| RTX5090 vs. H800 (LLaMA3.1-8B) | −53.3% deficit | **−14.1% deficit** |
| RTX5090 vs. H800 (Mistral-24B) | −125.7% deficit | **−20.8% deficit** |

**Comparison with Lossy (Marlin W8A16 FP8):** 0.194ms (ZipGEMM) vs. 0.143ms (Marlin) → 1.36× gap ≈ effective bit-width ratio (~11b vs. FP8).

### 4.8 Ablation / Sensitivity

| Factor | Finding |
|--------|---------|
| Offline compression | ~2.5 min (one-time, negligible over lifecycle) |
| Prefill decoupled overhead | ~4% at N=8192, ~2% at N=16384 |
| Training-oriented GPUs (A100/H800) | ZipGEMM not always beat cuBLAS (HBM bandwidth vs. ALU intensity mismatch) but still best compressed inference |
| nvCOMP BF16 | Lack of native support → exponent bitstream via rANS + custom kernel |
