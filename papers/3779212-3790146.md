# CPU-Oblivious Offloading of Failure-Atomic Transactions for Disaggregated Memory

**Venue:** ASPLOS 2026
**저자:** Cheng Chen*, Chencheng Ye*, Yuanchao Xu (UC Santa Cruz), Xipeng Shen (NCSU), Xiaofei Liao*, Hai Jin*, Wenbin Jiang*, Yan Solihin (UCF) — Huazhong University of Science and Technology

## 개요

CXL 기반 memory disaggregation은 compute server와 memory server를 분리하여 memory utilization과 cost-efficiency를 획기적으로 개선. 동시에 persistent memory (PM) landscape도 Optane PMEM 단종 이후 CXL-attached memory-semantic SSD (CMM-H)나 UPS-backed DRAM server로 전환 중.

**근본 문제**: Disaggregated memory에서 persistent memory transaction의 log persistence latency가 prohibitively 큼.

- CMM-H remote memory access: median 728.9ns latency [68]
- Persistent memory transaction의 persist barrier (sfence)는 CPU pipeline을 stall → Figure 1(a): synchronous write request는 CPU가 memory server로부터 response를 받을 때까지 대기
- Software-only approach (SpecPMT [66]): average 1.6× slowdown over local PM (Figure 2)

**Strawman solutions와 실패 이유**:

1. **Software-only optimization** (SpecPMT의 speculative logging): 단일 sfence로도 여전히 synchronous — CPU가 high-latency network를 통해 remote memory server까지 log data가 persist되기를 기다려야 함.

2. **CPU-centric hardware acceleration** (ASAP [6], HOOP [13]): CPU core modification 필요 → heterogeneous datacenter (AWS Graviton, Microsoft custom silicon 등)에서의 vendor-agnostic CXL 철학에 위배.

3. **Background thread persistence** (DudeTM [39]): Log를 local DRAM shadow memory에 복사 → 4-6× write amplification.

**Goal**: CXL-based disaggregated memory를 위한 failure-atomic transaction system, CPU modification 없이 persistence latency를 critical path에서 제거.

## 방법론

### 1. Asynchronous transaction model (Figure 5)

Fanmem의 소프트웨어 모델은 transaction lifecycle을 3단계로 분할:

- **Execution**: tx_begin(), data modification 직접 수행. Speculative logging — log record 생성 (new data values).
- **Offloading**: Log를 pre-allocated disaggregated memory region에 write → clwb → sfence. **Fanmem-enabled CXL switch가 sfence를 early acknowledgment로 변환**: log write request가 switch에 도달하는 즉시 ACK → CPU는 sfence를 retire하고 다음 작업 진행 가능.
- **Commit**: isPersisted(txid) polling → tx_commit(txid). Log durability 확인 후 commit.

이 모델은 application이 transaction의 execution 완료 후에도 다른 작업을 수행할 수 있게 하며, commit을 지연시켜 batching 효과를 얻음.

### 2. Sequential log + cursor mechanism (Section 3.2, Figure 6)

**Per-thread, append-only sequential log area**: Application thread는 physically contiguous한 disaggregated memory region을 log 전용으로 할당, 모든 log record를 sequential write.

**Hardware-managed cursor**: CXL switch 내 Fanmem hardware에 per-log-area cursor 유지. Cursor value는 "log area 시작부터 cursor address까지의 모든 log data가 remote memory server에 durable함"을 보장.

**isPersisted() check**: 단일 mmap'd load instruction으로 cursor 값 조회 → transaction의 last log byte address ≤ cursor 이면 persistence 보장. system call/interrupt 불필요, overhead negligible.

### 3. Hardware-assisted offloading (Figure 7, Section 3.3)

Fanmem은 CXL switch에 두 개의 경량 hardware component 추가:

**Log Area Table (LAT)** — log write 식별
- Partitioned design (Figure 8): Index Table (source port ID indexed) + multiple pLATs (range-based TLBs)
- pLAT entry: base physical address (4KB granularity) + size (in 4KB) + cursor ID
- Flow: Write request 도착 → source port ID로 Index Table lookup → pLAT index 획득 → pLAT에서 address range matching → hit = log write, miss = normal traffic
- Lookup latency: 6 cycles

**Log Write Status Table (LWST)** — in-flight log write tracking (Figure 9)
- Partitioned design: pLWST per log area
- pLWST entry: cursor (6B, oldest in-flight write의 base address) + sliding window (4B, 64-byte log entries의 상태를 2-bit encoding으로 tracking: empty/buffered/persisted)
- 3 operations:
  - **Insertion**: Cursor로부터 offset 계산 → bit-vector update → 즉시 ACK to compute server
  - **Persistence Confirmation**: Memory server로부터 write completion response → bit-vector "persisted"로 마킹 → cursor advancement across contiguous "persisted" entries
  - **Cursor Query**: mmap'd load → LWST returns current cursor value

**End-to-end flow** (Figure 7):
1. Write arrives from compute server
2. LAT lookup → log write 판별
3. Non-log write → transparent forward to memory server
4. Log write → LWST insertion + immediate ACK to CPU + forward to memory server
5. Memory server completion → LWST update + cursor advancement

## 핵심 기여

- **Fanmem**: CXL-based disaggregated memory를 위한 최초의 failure-atomic transaction system. Compute server, CXL switch, memory server failure에 대해 crash consistency 보장.
- **핵심 innovation**: Log persistence를 CXL switch로 offloading하여 asynchronous transaction model 구현. CPU modification 불필요 (CPU-oblivious) — any CXL-compliant CPU와 호환.
- **CXL-F에서 1.2×, CXL-S에서 1.7× avg throughput over SpecPMT**. Asynchronous scheme임에도 polling overhead 0.5-0.7%로 negligible.
- **Hardware cost**: 2,590 FFs + 3,361 LUTs + 594KB BRAM (ASIC: 1.4mm², 479mW). Switch resource의 미미한 부분만 사용.
- **Broader significance**: Persistent memory transaction을 disaggregated memory 환경에서 실용화하는 foundational mechanism. Speculative logging, redo logging 등 다양한 logging protocol에 적용 가능.

## 주요 결과

**Timestamp-based** protocol, SpecPMT [66]와 유사:

1. **Timestamping**: tx_end() 시 per-process timestamp 할당 (rdtscp or monotonic counter). Cross-server timestamp synchronization 불필요.
2. **Dependency tracking**: Execution order 기반 — 이전 timestamp의 모든 transaction에 dependency.
3. **Committing**: 각 thread가 completed-but-uncommitted transaction의 sorted list 유지. Globally oldest timestamp의 ready (isPersisted) transaction 찾음 → per-process commit timestamp를 atomically advance → 해당 timestamp 이하의 모든 ready transaction 일괄 commit.

**Batching**: 다수의 contiguous ready transaction을 한 번의 timestamp update로 commit 가능 (Figure 10).

**Recovery**: Per-process commit timestamp를 memory server에서 읽고, timestamp 이하의 committed log를 replay하여 consistent state로 복원. Volatile switch buffer의 in-flight data는 자연히 무시됨.
