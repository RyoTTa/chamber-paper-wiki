# RowArmor: Efficient and Comprehensive Protection Against DRAM Disturbance Errors

**Venue:** 
**저자:** 

## 개요

### RowHammer 위협 심화
- DRAM 공정 미세화로 셀 간 물리적 격리 감소 → disturbance error 급증
- **NRH (RowHammer Threshold)**: DDR3 139K → **LPDDR4 4.8K** [44] — 약 29배 감소
- NRH 변동성 극심: DDR4 모듈 15개 대상 측정 시 **8K~56K** (factor of 7) [89]
- 단일 칩 내에서도 행마다 최대 **16배 차이** (8K 행 vs. 128K 견디는 행)
- BER (Bit Error Ratio): **0.01%~8%** 범위, 트랜지스터 aging, data pattern 의존성 존재

### Emerging Attack Patterns
- **RowPress [57]**: row open time을 길게 유지 → 더 적은 activation으로 bit flip 유발 (NRH 우회)
- **Half-Double [18, 44, 49]**: 인접 행뿐 아니라 원거리 행까지 disturbance 전파
- **Marionette [7]**: row-coupling 효과로 non-adjacent 원격 victim 공격
- 기존 activation-count 기반 모니터링으로 탐지 불가

### 기존 Preventive 방식의 한계
- **TRR / RFM / PRAC** 모두 worst-case NRH 기준 → 불필요한 preventive action 다량 발생
- PRAC back-off: 이론상 **최대 94% DRAM 대역폭 소모** 가능 [16]
- Deterministic tracking 구조 (Graphene CAM, Hydra SRAM 32KB) → NRH 감소 시 area overhead 폭증
- Preventive action 자체가 side-channel / DoS 공격 벡터로 작용 [11, 84]
- Activation counter 업데이트가 DRAM timing parameter 변경을 강제 → 성능 저하 불가피 [42]

### ECC의 이중성
- DDR5: On-Die ECC 6.25% + Rank-Level ECC 25% → **총 32.8% redundancy**
- 기존 Chipkill-correct ECC: disturbance error는 multiple chip 동시 손상 → NRH **2~3배 증가에 그침** [47]
- 8-symbol redundancy QPC: detection capability ≈ **2×10^19 syndromes**, detection rate **99.999999%**
- **DUE (Detected but Uncorrectable Error)** → MCA/EDAC 통한 머신 체크 → SGX-Bomb 식 시스템 정지 [34], 데이터센터 서버 교체 프로토콜 [74] 발동 → **DoS 공격으로 이어짐**
- SDC (Silent Data Corruption) 확률은 극히 낮으나 (QPC: 2×10^-9), correction 실패 시 DUE → DoS

## 방법론

RowArmor는 4가지 원칙 기반 reactive defense: **Confine → Correct → Obfuscate → Scrub**

### 1. Octet Scrambling: Error Confinement

**목표**: 단일 aggressor 행의 피해를 **access당 최대 1개 octet (8-bit)** 으로 제한

**Octet 정의**:
- DDR5 BL=16, 하나의 DQ가 16-bit 전송 (8-bit × 2 MATs)
- 8-bit half-burst = 1 octet, transfer block = **80 octets** (64 data + 16 redundancy)
- DQ별 2개 octet, 10 chips × 4 DQ = 40 DQ → 80 octets

#### 1a. DAS (DQ Address Scrambling)
- DRAM row decoder 내 local decoder 앞단에 **4개 scrambling block (F0~F3)** 삽입
- Modular multiplication: `MA_i = (k_i × DA) mod 1024`
- **계수 제약조건**:
  - (1) 홀수여야 1:1 mapping 보장
  - (2) sub-channel 내 DQ 간 unique
  - (3) `k_i ≠ (-k_j) mod 1024` → +1, -1 victim row 충돌 방지
- **Chip-level diversity**: ×4 ECC DIMM sub-channel = 10 chips → DQ별 10개 unique multiplier 할당
- **예시 (Table 2)**: DQ0/Chip0 = DA×1, DQ1 = DA×5, DQ2 = DA×9, DQ3 = DA×13, ...
- Antifuse hardwiring → runtime multiplexer 불필요 (post-package repair [41] 방식 재활용)
- tRCD 증가: 1 DRAM cycle

#### 1b. SWLP (Sub-WordLine Permutation)
- **문제**: 동일 DQ 내 두 MAT는 decoder/SWLD 공유 → DAS만으로 두 octet 분리 불가
- **해결**: SWLD 이후 signal permutation 적용
- `i번째 SWL → ((3×i) mod 8)번째 LWL` (multiplier 3, 8과 coprime → 1:1 mapping)
- Modular multiplication 기반 배선 변경 → **추가 하드웨어 비용 0**
- **DAS-SWLP 통합 제약**: ∀i,j, `(k_i mod 8) ≠ (k_j mod 8)`
- 확장성: MAT 4개/DQ → 왼쪽 2 MAT 직결, 오른쪽 2 MAT에 SWLP

**효과 (예시)**:
- `DA_Aggr` hammering 시 인접 `DA_Aggr±1` victim rows
- DAS+SWLP로 octet별 물리적 주소 분산 → 공격자는 최대 **2개 octet error/request** 유발 (double-sided)
- Half-Double 원거리 victim도 octet scrambling으로 confinement 유지

**Strawman — DAS-only**: 동일 DQ 내 두 octet 동시 corruption 가능 → ECC correction 능력 초과 위험
**Strawman — Chip-level only (Cube [47])**: DQ-level 미분화 → chip 내 모든 DQ 동일 mapping → multi-aggressor 정렬 가능

### 2. OOC ECC (Octuple-Octet Correcting ECC)

**Codeword 구조**:
- Bamboo ECC [43] 기반, DDR5 BL=16에 맞춰 DQ당 2 symbols 할당 (Figure 4)
- **Codeword = 64 data symbols + 16 parity symbols = 80 symbols** (symbol = octet = 8-bit)
- Reed-Solomon code → **최대 8개 octet error correction**

**Correction Capacity**:
- 1~8개 aggressor → 모든 overlapping error를 octet 단위로 교정 → **100% protection**
- Single-aggressor: octet scrambling으로 2개 error → OOC가 여유 있게 교정
- 8-aggressor: 8개 overlapping octet error → OOC limit, 정확히 교정

**Correction Validation 메커니즘**:
- **문제**: OOC의 강력한 correction이 random error에 대해 miscorrection (SDC) 위험 증가
- **트리거 조건**: correction 결과 error pattern이 아래 중 하나를 초과
  - (1) 1 chip 내 8개 octet (Chipkill)
  - (2) 2 pins × 4 octet
  - (3) 3개 독립 octet
- **판별 절차**:
  1. MRR (Mode Register Read) → OD-ECC error counter snapshot
  2. 동일 row 내 추가 column read
  3. MRR 재실행 → counter 증가 확인
  4. Counter 증가 = disturbance (같은 row 내 전파) → full correction **수락**
  5. Counter 불변 = random error → restricted correction으로 fallback, **DUE 보고**
- SDC 확률: **5×10^-19** (수학적 분석; Monte Carlo로는 측정 불가)

**지연시간**:
- 전체 block 수신 후 decoding 시작 필요 → DDR PHY 8:1 deserialization으로 buffering delay가 memory controller cycle 1회로 제한 [22]
- Read latency: +1 controller cycle

### 3. RAO (Row Address Obfuscation)

**목표**: 9+ aggressor 공격에 대해 overlapping error 확률을 통계적으로 불가능하게 만듦

**구현**:
- **Feistel cipher** 기반 주소 암호화 (XOR S-box, 효율적 HW 구현)
- OS가 boot 시 **per-bank random key** provisioning → memory controller 내 암호화
- 암호화는 command scheduling과 병렬 수행 → **추가 latency 없음**
- Physical address → DRAM row address mapping을 randomized one-to-one permutation으로 변환

**효과 분석 (Figure 5, Appendix A)**:
- R=128K rows/bank 가정
- **9-aggressor → 4×10^-27** (9개 overlapping error 확률)
- **128-aggressor → 6×10^-14**
- OOC ECC limit (8-error)을 넘는 9-error overlap이 통계적으로 infeasible

**Reverse-engineering 저항성**:
1. Reactive correction은 infrequent & random → timing side-channel 신호 약함
2. ECC correction delay: 수 cycle (vs. refresh: 수백 cycle)
3. 160개 octet 분산 → error-timing correlation 추출 난이도 극대화
4. Per-bank key rotation 가능 (scrub trigger 시)

**Strawman — No obfuscation**: 공격자가 address mapping 학습 → 특정 victim에 대한 다수 aggressor 정렬 가능
**Strawman — Controller-only obfuscation**: Cube/RAMPART처럼 scrambling+obfuscation 결합 → 유연성 부족, overhead 증가

## 핵심 기여

### Core Contributions
1. **Reactive DRAM disturbance defense 패러다임 제시**: preventive tracking/refresh 대신 error 발생 후 ECC로 대응 → NRH 감소에 따른 overhead 증가 문제 해결
2. **Octet scrambling (DAS+SWLP)**: 기존 chip-level scrambling을 DQ/octet granularity로 세분화 → single-aggressor 피해를 8-bit로 confinement
3. **OOC ECC + Correction Validation**: 8-symbol correction으로 8-aggressor까지 100% 보호, SDC 5×10^-19로 random error 안전성 유지
4. **Guardband scrubbing**: partial correction margin을 활용한 proactive error accumulation 차단 → 10% BER 극한 상황에서도 saturation으로 DoS 확률 1.4×10^-11 유지

### Broader Significance
- **"방어를 tracking 문제에서 error-tolerant data path 문제로 재구성"** — 공격 패턴 변화 (RowPress, Half-Double, Marionette)에 pattern-agnostic하게 대응
- DDR5의 막대한 ECC 투자 (32.8% redundancy)를 security 목적으로 활용 → 추가 storage overhead 거의 없음
- Performance overhead **0.7% 이하**, NRH 독립적 → 미래 DRAM 세대에도 적용 가능
- Hardware cost: MC <0.00001%, DRAM <0.005% — PRAC 대비 수십 배 작음
- **한계**: 8-aggressor 초과 시 확률적 보호로 전환, bank scrubbing 0.1초 latency, DRAM/controller 양측 수정 필요

## 주요 결과

**목표**: Error accumulation을 uncorrectable 수준 도달 전에 차단

**동작 방식**:
- **Trigger threshold**: OOC ECC correction 중 **3개 이상 correctable error** 감지
- OOC correction 용량 8개 중 3개 → **guardband = 5개 error** margin 확보
- Scrubbing: 해당 bank 전체를 read-correct-write sweep

**분석 모델 (Figure 6)**:
- 공격 모델: 매 라운드 1개 aggressor 추가 + M개 normal row access
- `P_3-8(R, A-1)`: A-1개 aggressor 공격 후 3~8개 error 누적 확률
- `P_rnd_scrub = 1 - (1 - P_3-8)^M`: 라운드 내 scrubbing trigger 확률
- `P_rnd_9err = R × P(R,A-1,8) × 2(80-8)/(R-(A-1))`: 새로운 aggressor가 9-error overlap 유발 확률
- Bank state: OK → Scrub (terminal) or UE (terminal)

**결과**:
- Without scrubbing: 256 aggressor → **4.4×10^-6** UE 확률
- M=128 (보수적): 168 aggressor → **3.8×10^-9** plateau
- M=2048: 87 aggressor → **6.2×10^-12** plateau
- Scrubbing 비용: 32Gb DDR5-6400 bank scrub ≈ **0.1초**

**Escalation 전략**:
1. **OS throttling**: MCA register 통해 error address 모니터링 → 의심 thread에 Intel MBA [32] 적용
2. **Key rotation** (last resort): persistent attack → bank 일시 정지, per-bank obfuscation key 갱신, permutation cycle 기반 data migration (row-size buffer 1개로 swap 연산)

|----------|----------|-----------|------------|

| PARA [48] | Preventive/Probabilistic | p=0.079 | Victim refresh |

| Cube_PARA [47] | Reactive+Preventive | p=0.062 + chip scramble | Victim refresh |

| PRAC-4 [5] | Preventive/Deterministic | Per-row counter (DRAM) | RFM×4 |

**공통**: 모든 baseline 및 RowArmor는 (수정된) AMD Chipkill ECC 사용 (DDR5 ECC-DIMM에 맞춰 (8+2) 8-bit symbols). RowArmor는 OOC ECC로 대체.

**RowArmor 설정**: tRCD +1 DRAM cycle, read latency +1 memory controller cycle. OOC-only와 OOC+Scrubbing 두 구성 평가.

### Security: Targeted Attack (Figure 7a, 8a)

**Threat model**: 64B block corruption without detection, 24시간 공격, 600K activations/32ms window

**BER=0.01% 결과**:
| Aggressors | Baseline | PARA | SRS | Cube_SRS | RAMPART | **RowArmor** |
|-----------|----------|------|-----|----------|---------|-------------|
| 1 | 2.3×10^-9 | ~10^-19 | ~10^-25 | ~10^-29 | ~10^-28 | **0** (8개까지 100% 보호) |
| 256 | — | — | — | — | — | **3×10^-57** |

- Baseline도 AMD Chipkill detection으로 2.3×10^-9 → targeted attack 이미 어려움
- Cube_SRS: 1.1×10^-29, RAMPART: 3.3×10^-28
- RowArmor: 8-aggressor까지 **절대적 안전**, 256-aggressor에도 **3×10^-57**

**BER=10% 결과**:
- 128-aggressor targeted: **8.4×10^-37** (RowArmor with scrub)
- Cube, RAMPART: BER 증가 시 correction에 의존 → 보안 약화
- RowArmor: scrubbing이 high BER에서 더 효과적 (error 관측 용이)

### Security: DoS Attack (Figure 7b, 8b)

**BER=0.01% 결과**:
| Aggressors | Baseline | PARA | SRS | RAMPART | **RowArmor** |
|-----------|----------|------|-----|---------|-------------|
| 1 | ~100% | 3×10^-8 | ~10^-12 | ~10^-18 | **0** |
| 16 | — | — | — | — | **1×10^-43** |
| 256 | — | — | — | — | **9×10^-32** |

- Baseline: single-aggressor로도 거의 100% DoS 성공
- RowArmor: 8-aggressor까지 DoS 완전 차단

**BER=10% 결과**:
- 128-aggressor DoS: **1.4×10^-11** (RowArmor with scrub)
- Scrubbing 없는 OOC-only: BER=10%에서 RAMPART보다 취약 (frequent error → correction 부담)
- Scrubbing 결합 시 가장 강력한 보호 → 64 aggressor 부근에서 saturation

**NRH=2048 기준 Normalized Performance**:

| Scheme | Rate | Mix-High | Mix-Rand |
|--------|------|----------|----------|
| PARA | ~0.98 | ~0.96 | ~0.97 |
| SRS | ~0.96 | ~0.92 | ~0.95 |
| Cube_SRS | ~0.94 | ~0.88 | ~0.93 |
| Graphene | ~1.00 | ~0.99 | ~1.00 |
| ABACuS | ~0.99 | ~0.97 | ~0.99 |
| PRAC | ~0.97 | ~0.93 | ~0.96 |
| **RowArmor** | **~0.993** | **~0.993** | **~0.994** |

**NRH 감소에 따른 변화**:
- RAMPART: 가장 빠른 저하 (group-based monitoring → 불필요 refresh 증가)
- PRAC: NRH≥512에서 counter-update timing overhead dominant, NRH≤256에서 RFM idle dominant
- SRS/Cube_SRS: row-swap 비용이 가장 큼 → 최악 성능
- **RowArmor: NRH와 무관하게 ~0.7% 성능 저하로 일정** (Mix-High 기준)
- Graphene: 성능 우수하나 CAM area prohibitive + deferred refresh로 emerging attack 취약

### Reliability (Table 4)

**Monte Carlo simulation (10^8 trials/시나리오)**:

| Error Scenario | AMD Chipkill | Bamboo QPC | **OOC (RowArmor)** |
|---------------|-------------|------------|-------------------|
| 1 bit/1 pin/1 chip | 100% CE | 100% CE | 100% CE |
| 2× 1 bit | CE 9.89%, DUE 89.63%, SDC 0.49% | CE 100% | **CE 100%** |
| 3× 1 bit | CE 0.96%, DUE 97.68%, SDC 1.36% | CE 76.76%, DUE 23.24% | **CE 12.42%, DUE 87.58%** |
| 1 chip + 1 bit | CE 96.87%, DUE 3.13% | CE 10.00%, DUE 89.99% | **CE 12.42%, DUE 87.58%** |
| 1 chip + 2 bits | CE 0%, DUE 99.39% | CE 1.00%, DUE 99.00% | **CE 1.59%, DUE 98.41%** |
| 1 chip + 1 chip | DUE 100% | DUE 100% | DUE 100% |
| Chipkill (rank) | DUE 100% | DUE 100% | DUE 100% |

- OOC: 3개 1-bit error까지 partial correction 가능 (AMD Chipkill은 2개에서도 SDC 0.49%)
- SDC: OOC의 **5×10^-19** vs AMD Chipkill **2×10^-9**
- Chipkill 보호는 세 방식 모두 유지

### Ablation: Scrubbing 효과

| Configuration | BER=0.01%, 256aggr | BER=10%, 128aggr |
|--------------|-------------------|-------------------|
| OOC only | Targeted: ~3×10^-57, DoS: ~9×10^-32 | Targeted: ~10^-25, DoS: ~10^-4 |
| OOC + Scrubbing | Targeted: ~3×10^-57, DoS: ~9×10^-32 | Targeted: **8.4×10^-37**, DoS: **1.4×10^-11** |

- Low BER: scrubbing 효과 제한적 (error accumulation 자체가 드묾)
- High BER: scrubbing이 결정적 역할 — error를 조기에 관측·차단하여 saturation 유도

## 구현

- **RTL 합성 도구**: Synopsys Design Compiler [50], UMC 28nm library
- **OOC ECC**: Bamboo ECC [43] 기반 확장 — RS code, 64+16 symbol codeword
- **RAO**: Feistel cipher 하드웨어 구현 — XOR S-box, 명령어 스케줄링과 파이프라인 병렬화
- **DAS**: Row decoder 내 constant multiplication logic — antifuse hardwiring (post-package repair 재활용)
- **SWLP**: Modular multiplication 기반 배선 permutation — 추가 gate 없음
- **Correction Validation**: OD-ECC error counter + MRR command 프로토콜 활용
- **Simulator**: McSimA+ 기반 수정 — SPEC CPU2017 Pintool trace, SimPoint clustering
- **Artifact**: GitHub (https://github.com/scalable-arch/RowArmor), Excel 보안 분석 + C++ 신뢰성 Monte Carlo + McSimA+ 성능 시뮬레이션
