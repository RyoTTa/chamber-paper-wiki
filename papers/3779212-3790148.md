# CREST: High-Performance Contention Resolution for Disaggregated Transactions

**Venue:** ASPLOS '26 (Vol. 2), March 22–26, 2026, Pittsburgh, PA, USA
**저자:** Qihan Kang, Mi Zhang, Patrick P. C. Lee, Yongkang Hu

## 개요

Memory disaggregation은 compute와 memory 자원을 분리하여 resource scaling에 유리하지만, **high-contention workload**에서는 기존 disaggregated transaction system이 심각한 성능 저하를 겪음.

**Motivating data (FORD, Motor 기준):**
- TPC-C: warehouses 80→20로 contention 증가 시 FORD throughput 71.2% 감소 (400→105 KOPS), Motor 57.3% 감소
- SmallBank (Zipfian θ=1.22): FORD 81.7%, Motor 80.1% throughput 감소
- TPC-C에서 abort rate가 Motor 85.2%, FORD 75.9%에 달하며, 이 중 **false abort rate**는 Motor 44.1%, FORD 40.7% (Figure 3)
- Motor commit latency breakdown: execution latency가 TPC-C에서 119.8%, SmallBank에서 152.9% 증가 (Figure 4)

**Root causes:**
1. **Record-level concurrency control의 false conflict:** 같은 record의 서로 다른 column field에 접근하는 transaction들이 불필요하게 abort됨. 예: TPC-C warehouse table에서 NewOrder는 name/location column만 읽고, Payment는 balance column만 update함에도 record-level lock으로 충돌.
2. **긴 blocking time:** 기존 시스템은 transaction이 commit될 때까지 lock을 유지하며, disaggregated 환경에서는 commit에 여러 network round-trip이 필요해 blocking이 증폭됨.

## 방법론

CREST는 세 가지 핵심 기술로 구성됨 (Figure 5):

### 1. Cell-level Concurrency Control (§4)

**목적:** Record-level false conflict 제거. Cell = record 내 개별 column field.

**Record structure (Figure 6):**
- 각 cell은 64B cacheline 내에 cell version(EN 2B + TScommit 6B) + cell value를 co-locate → single RDMA READ/WRITE로 atomic access
- **Record header** (one cacheline): TableID(8B), Key(8B), Lock(8B), EN array (각 cell당 2B epoch number)
- EN array는 모든 cell의 epoch number를 연속 배치 → single RDMA READ로 multi-cell validation 가능
- Lock field: cell당 1bit, 8-byte에 64개 cell 지원

**Multi-cell locking:** masked-CAS primitive로 여러 cell lock을 단일 RDMA call로 atomic modify.
- 예: 2번째, 4번째 cell lock → CAS mask = `0101...000`

**Multi-cell validation:** EN array를 record header에서 한 번에 읽어 validation. 2-byte EN overflow 방지를 위해 time threshold (65,536μs) 사용 — overflow 전에 전체 record를 읽어 TScommit 기반 validation.

**Conflict resolution (Figure 7):**
- **Intra-cell:** cacheline atomicity로 보장
- **Inter-cell:** Lock + EN array snapshot을 하나의 cacheline RDMA READ로 획득 → (i) read cell의 lock이 다른 txn에 의해 획득되지 않았고, (ii) header의 EN과 cell snapshot의 EN이 일치하면 consistent snapshot

**제약:** header cacheline(64B) 내 EN array(2B×N) + Lock(8B) → 최대 20개 cell, Lock은 최대 64개. 20개 초과 column은 하나의 large cell로 통합 (false conflict 재발생 가능).

### 2. Localized Execution (§5)

**목적:** 같은 compute node 내 transaction 간 blocking time 감소. Uncommitted result를 local cache로 expose.

**Record cache (Figure 8):**
- Compute node별로 record cache 유지 (concurrent hash map 기반)
- Local object 구조: `reference counter` (readers/writers 변수), `epoch array` (record header EN array 복사본), `version list` (uncommitted version linked list)
- Cache mode: read-write (cell lock 획득 필요) / read-only
- Admission: 최초 access 시 record cache에 생성, uncached record는 memory pool에서 fetch. 중복 fetch 방지를 위해 하나의 coordinator만 admission 수행.

**Transaction dependency tracking:**
- 각 transaction은 unique txn ID를 가지며, 접근한 local version에 자신의 ID를 embedding
- 다른 txn이 local version에 접근 시, 해당 txn ID를 자신의 dependency list에 추가 → commit phase에서 dependency check

**Pipelined execution (Figure 9, 10):**
- Transaction workflow를 execution block으로 분할 (key dependency 단위로 group)
- **Inner-block:** 2PL 적용. Local lock은 (TableID, Key) ascending order로 획득 (deadlock 방지). Block 완료 시 lock release.
- **Block ordering:** 각 txn은 첫 block에서 monotonically increasing counter로 TSexec 할당. Record write 시 local version에 TSexec tagging. 이후 txn이 더 큰 TSexec의 version을 읽으면 reverse order 감지 → abort.

### 3. Parallel Commits (§6)

**목적:** Network round-trip 최소화하며 correctness 보장.

**Validation:**
- Read-only set의 모든 record에 대해, record cache에서 읽은 EN과 memory pool의 최신 EN 비교 → 불일치 시 abort
- Dependency check: dependent transaction이 abort된 경우 현재 txn도 abort
- 통과 시: committable로 marking, commit timestamp(TScommit) 할당

**Dependency-tracking redo-logging:**
- Log entry: txn ID, modification logs, dependent txn ID list
- Fixed-size log segment를 coordinator별로 할당, RDMA WRITE로 append
- Txn은 log entry가 memory pool에 persist된 시점에 committed로 간주

**Last-writer-wins:**
- Reference counter의 writers 변수로 마지막 writer 판별
- Committable 확인 후 writers decrement, 0 도달 시 해당 coordinator가 last writer
- Last writer만 memory pool에 record update + lock release 수행 (ordered RDMA WRITE + masked-CAS batch)

## ACID 보장 및 Overhead (§7)

**Atomicity:** Redo-logging + last-writer-wins. 동일 cell에 대한 concurrent update는 highest TScommit 승리.
**Consistency:** Cell-level conflict detection (intra-cell cacheline atomicity + inter-cell snapshot verification).
**Isolation:** TSexec(block ordering) + EN(global ordering) + 2PL(local serializability).
**Durability:** Redo-log entry RDMA WRITE 후 committed marking. Primary-backup replication.

**Space overhead (Table 1):**
- TPC-C: CREST 20.65% (without padding) / 41.30% (with padding), vs FORD 7.49%/19.61%, Motor 58.75%/63.92%
- SmallBank: CREST 42.19%/100%, Motor 318.75%/400%
- Compute node memory: TPC-C average 2.48 KiB (FORD 1.96, Motor 3.99)

**RDMA communication (Table 2):**
- CREST는 FORD/Motor와 동일한 수의 RDMA RTT (execution: READ or masked-CAS+READ, validation: READ, commit: WRITE+masked-CAS)
- Localized execution은 일부 READ를 skip하여 RTT 감소

## 핵심 기여

CREST는 disaggregated memory architecture에서 **high-contention transaction workload**를 효율적으로 처리하기 위한 세 가지 기법을 제안:
1. **Cell-level concurrency control**로 false abort 제거 (record 대신 column field 단위)
2. **Localized execution**으로 compute node 내 uncommitted result 공유로 blocking time 감소
3. **Parallel commits**로 dependency-preserving commit 병렬화

FORD/Motor 대비 최대 **1.92× throughput**, skewness가 높을수록 gain 증가 (최대 3.64×). C++ 14K LOC로 구현, 오픈소스 공개. RDMA one-sided primitive 기반의 실용적인 접근으로, production 환경(veDB, PolarDB-MP, GaussDB)에서도 적용 가능한 설계.

## 주요 결과

### 방법론

| 항목 | 내용 |
|------|------|
| **Testbed** | 3 compute nodes + 2 memory nodes, Intel Xeon Platinum 8260 (48 cores), 100 Gbps Mellanox ConnectX-5 InfiniBand |
| **Workloads** | TPC-C (40 warehouses default), SmallBank (100K accounts, Zipfian θ=0.99), YCSB (1M records, 4 cells×40B, θ=0.99) |
| **Baselines** | FORD, Motor (open-source prototypes) |
| **Config** | 40 threads per CN × 2 coroutines = 240 total coordinators (default) |
| **Metrics** | Throughput (KOPS), average/P50/P99/P999 latency (μs) |
| **구현** | C++ 14K LOC, coroutine 기반, doorbell batching + selective signaling |

### Throughput (Figure 11)

240 coordinators 기준:
- **TPC-C:** CREST 743.7 KOPS → Motor 대비 **1.92×**, FORD 대비 **2.76×**
- **SmallBank:** CREST → Motor 대비 **1.46×**, FORD 대비 **1.89×**
- **YCSB:** CREST → Motor 대비 **1.85×**, FORD 대비 **2.58×**
- Scalability: CREST는 144 coordinators까지 throughput 증가, FORD는 72, Motor는 96에서 peak

### Latency (Figure 12, 13)

240 coordinators:
- **Average latency:** CREST가 FORD 대비 41.1-62.6%, Motor 대비 17.7-44.4% 감소
- **P50 latency:** YCSB에서 CREST가 FORD 대비 45.8%, Motor 대비 47.2% 감소
- **P99 latency:** TPC-C에서 CREST가 Motor 대비 15.2%, FORD 대비 33.7% 감소; SmallBank에서 Motor 대비 36.7%, FORD 대비 42.1% 감소

### Factor Analysis (Figure 15, high skewness)

Baseline 대비 normalized throughput:
- **+Cell-level:** TPC-C +65.9%, YCSB +46.6%, SmallBank 변화 없음 (single-column)
- **+Localized execution (+parallel commits):** TPC-C +48.9%, SmallBank +78.1%, YCSB +104.6% 추가 향상

### Skewness Sensitivity (Figure 16)

최대 skewness에서 CREST throughput = FORD 대비:
- TPC-C **3.64×**, SmallBank **2.91×**, YCSB **4.54×**
- Motor 대비: TPC-C 2.72×, SmallBank 2.05×, YCSB 3.02×
- CREST의 throughput gain은 skewness 증가에 따라 커짐 (TPC-C: moderate skewness 172.4% → high skewness 264.4% over FORD)

### Write Ratio Impact (Figure 18, YCSB θ=0.99)

- Write ratio ≥ 50%: CREST가 FORD 대비 180.9%, Motor 대비 105.1% 높은 throughput
- Write ratio 25%: Motor 대비 16.9% 우위
- Read-only: Motor가 CREST보다 12.8% 우위 (Motor는 MVCC로 read-only validation skip; CREST는 local cache management overhead)

### 다른 Sensitivity

- **Records count (Figure 17):** N 증가 시 CREST gain 증가. YCSB θ=0.99에서 N=1일 때 Motor 대비 25.7%, N=4일 때 102.3%.
- **Low skewness (θ=0.1):** 모든 시스템 유사한 성능. CREST는 state-of-the-art와 동등.

## 구현

- **언어/규모:** C++ 14K LOC (from scratch)
- **Key components:**
  - Coroutine 기반 coordinator (thread당 2개)
  - Concurrent hash map 기반 record cache
  - Doorbell batching + selective signaling (CPU overhead 감소)
  - Pre-registered memory region에서 local version 할당
- **Open source:** https://github.com/adslabcuhk/crest (Apache 2.0)
