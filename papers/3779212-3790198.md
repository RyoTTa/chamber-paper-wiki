# PACT: A Criticality-First Design for Tiered Memory

**Venue:** 
**저자:** Hamid Hadian, Jinshu Liu (equal contribution), Hanchen Xu (equal contribution), Hansen Idden, Huaicheng Li (Virginia Tech)

## 개요

기존 tiered memory system은 **access frequency(hotness)**를 page placement의 기준으로 삼는다: 자주 접근되는 page가 performance-critical하다고 가정한다. 그러나 이 가정은 근본적으로 틀렸다:

- **Frequency ≠ Criticality:** 동일한 access frequency의 두 page라도, 하나는 pointer chasing(low MLP, serialized)으로 full memory latency에 노출되고, 다른 하나는 array streaming(high MLP, concurrent)으로 latency가 amortize된다. 전자는 높은 CPU stall을 유발하고, 후자는 minimal impact를 준다.
- **실측 (Figure 1):** Masim, GUPS, tc-twitter에서 동일 access frequency의 page 간 PAC(Per-page Access Criticality, CPU stall cycles) 차이가 최대 **65배** (tc-twitter: 7~460 cycles). GUPS는 4회 access된 page들 간에도 30~120 cycles로 4배 차이.

SoarAlto는 Amortized Offcore Latency(AOL)로 frequency의 한계를 인식했으나, coarse-grained object 단위 + offline profiling에 그친다. Colloid/Nomad/NBT/TPP/Memtis는 모두 hotness 기반으로, 과도한 migration(최대 수천만 건)에도 성능 향상이 제한적이다.

**핵심 질문:** 각 page access가 실제로 CPU stall에 얼마나 기여하는지 online, page-granular하게 정량화할 수 있는가?

## 방법론

### 1. PAC (Per-page Access Criticality) — 핵심 Metric

PAC는 각 page의 slow-tier access가 유발하는 **CPU stall cycles**를 정량화한다.

#### 1.1 Per-Tier Stall Modeling (Equation 1)

96개 workload × 3가지 latency configuration (DRAM 90ns, NUMA 140ns, emulated CXL 190ns) 분석에서 도출:

```
LLC-stalls = k × (LLC-misses / MLP)
```

- `k`: tier-specific coefficient (latency, memory controller queuing delay 등 포함). Pearson correlation **> 0.98** (Figure 2).
- `MLP`: Memory-Level Parallelism. 높을수록 per-access stall 감소.
- 이 공식은 bandwidth-bound scenario에서도 일반화: contention으로 인한 effective latency 증가가 k의 증가로 반영됨.

#### 1.2 Per-Tier MLP 측정 — CHA TOR Queue 활용

Intel CPU의 Caching and Home Agent(CHA)는 core와 offcore(DRAM/CXL) 사이에 위치하여 모든 memory request를 관찰 가능:

- **TOR_OCCUPANCY (T1):** TOR(Table of Requests) queue의 in-flight request 수.
- **TOR_OCCUPANCY_COUNTER0 (T2):** TOR가 non-empty였던 cycle 수.
- `MLP = ΔT1 / ΔT2`: active cycle당 평균 in-flight request 수.

L2MLP(system-wide)와 TOR-MLP가 높은 상관관계를 보이며 (Figure 3a), TOR-MLP는 per-tier 구분이 가능하다는 장점이 있다.

#### 1.3 MLP Phase Stability → Uniform Attribution

Workload는 quasi-stationary phase로 실행되며, MLP는 **tens of milliseconds** 동안 안정적 (Figure 3b). 20ms sampling window 내에서는 모든 access가 유사한 MLP를 경험한다고 가정 → proportional attribution 가능:

```
PAC[p] = total_slowtier_stalls × (A_p / A_t)
```

- `A_p`: page p의 access count (PEBS sampling).
- `A_t`: total sampled accesses.
- Exponential moving average(α ∈ [0,1])로 temporal aggregation (default α=1.0, no cooling).

### 2. PAC Sampling Algorithm (Algorithm 1, every 20ms)

1. T1, T2 counter delta로 per-tier MLP 계산 → Equation 1으로 slow-tier stalls(S) 추정.
2. Intel PEBS로 page access sampling (default rate: 1-in-400 events).
3. `S_p = S × (A_p / A_t)`로 각 page에 stall proportionally distribute.
4. `PAC[p] = α × PAC[p] + S_p`로 update (hash table 저장).

메모리 overhead: 25 bytes/page → 0.6% (4KB page 기준). 2개 전용 thread (PEBS processing + migration).

### 3. Eager Demotion Policy (Algorithm 2)

Fast-tier 공간 부족으로 high-PAC page의 promotion이 지연되는 것을 방지:

- `N_demoted ≥ N_promoted + m` 유지 (m: demotion aggressiveness, default 0).
- Promotion rate가 높으면 m 증가 → proactive demotion.
- Linux kernel의 (MG)LRU list에서 LRU page 선택적 demotion.
- 초기 execution에서 fast-tier utilization이 낮을 때 aggressive하게 headroom 확보, utilization 증가에 따라 점진적 완화.

## 핵심 기여

**핵심 contribution:**

- **Performance criticality를 tiered memory의 first-class design principle로 격상.** Frequency-hotness를 proxy로 사용하는 기존 패러다임을 근본적으로 전환.
- **PAC metric:** 단 4개의 CPU performance counter로 online, page-granular, per-tier criticality 추정이 가능함을 입증.
- **Criticality-centric policies:** Eager demotion + adaptive binning으로 극도로 skewed된 PAC 분포에서도 안정적 migration 수행.

**정량적 성과:**
- 최대 61% performance improvement (best baseline 대비).
- **Up to 50× fewer migrations** (Colloid vs. PACT in 1:1 bc-kron: 11M vs. 800K).
- 12/13 workloads에서 best or near-best. gpt-2에서는 유일하게 NoTier 능가.
- Bandwidth contention 및 mixed access pattern에서도 robust.
- Soar(offline) 대비 online-only로 평균 3.3%만 뒤짐.

**Broader significance:** "Frequency = Criticality"라는 암묵적 가정의 해체가 tiered memory 설계 공간을 근본적으로 재편. MLP-aware stall attribution은 향후 multi-tenant CXL 환경, latency-weighted PEBS integration 등으로 확장 가능. CXL 3.2의 CHMU(CXL Hotness Monitoring Unit)와 결합 시 더욱 정확한 PAC 추정이 가능할 것으로 전망.

## 주요 결과

PAC 분포의 extreme skewness에 대응하기 위한 adaptive binning:

#### 4.1 Freedman-Diaconis Binning

Optimal bin width 결정:

```
W = 2 × (Q3 - Q1) / ∛n
```

- Q1, Q3: PAC 분포의 1사분위, 3사분위수 (Reservoir sampling으로 추정).
- n: tracked page 수.
- Outlier에 robust하고 integrated mean squared error 최소화.

#### 4.2 Reservoir Sampling

전체 page의 PAC 값을 정렬하는 overhead를 피하기 위해, fixed-size Reservoir(100 entries)로 uniform sampling:

- 첫 100개 page는 직접 저장.
- 이후 page는 `k/n` 확률로 Reservoir entry 대체 → online adaptation 가능.

#### 4.3 Scaling Optimization

Skewed/bursty workload에서 bin collapse 방지:

- `N_page / N_c > T_scale` → bin width 2배 확장 (spread pages).
- `N_page / N_c < T_scale` → bin width 1/2로 축소 (복원).

Highest-priority bin은 전체의 top 1-5%로 유지 → 안정적 promotion candidate 공급.

## 평가

### Methodology

| 항목 | 내용 |
|------|------|
| Testbed | Dual-socket Intel Skylake (Xeon 2.2GHz ×10-core), 96GB DDR4/socket, CloudLab |
| DRAM latency | 90ns (local), 140ns (NUMA cross-socket) |
| CXL emulation | NUMA cross-socket + uncore frequency reduction → **190ns** |
| Fast:slow ratios | 8:1, 4:1, 2:1, 1:1, 1:2, 1:4, 1:8 |
| Baselines (7) | Soar, Alto, Colloid, Nomad, TPP, Memtis, NBT (Linux NUMA Balancing Tiering) |
| Workloads (13) | GAPBS (bc-kron, bc-urand, bc-twitter, pr-kron, pr-twitter, tc-twitter, sssp-kron, cc-kron), GPT-2 inference, Redis/YCSB-C, Silo, SPEC CPU 2017 (603.bwaves, 631.deepsjeng, 657.xz) |
| Page sizes | 4KB, 2MB THP |
| Metrics | Normalized slowdown vs. DRAM-only baseline |

### bc-kron (4KB, Figure 4, Table 2)

| Baseline | PACT slowdown | Baseline slowdown | Promotions (PACT vs. Baseline) |
|----------|--------------|-------------------|-------------------------------|
| Colloid (2nd best) | 20-37% | 26-59% | **2.1-10.4× fewer** |
| NBT | 20-37% | 27-62% | 1.2-9.6× fewer |
| Memtis | 20-37% | 30-90% | - |
| Nomad | 20-37% | 100%+ | - |
| TPP | 20-37% | 800%+ | - |

Colloid는 1:1에서 11M, 1:8에서 30M promotions 수행. PACT는 800K (7×, 20× 적음).

### bc-kron (THP, Figure 5)

THP에서도 모든 baseline을 능가. Memtis(THP-aware) 대비 1-19% 향상. Colloid/NBT는 THP에서 variance 크고 성능 저하.

### All 12 Workloads (1:1, Figure 6)

- PACT가 12개 중 10개 workload에서 best.
- gpt-2: **PACT만 NoTier를 outperform** (27% vs. 51% Colloid, 49% Nomad slowdown).
- 603.bwaves, 631.deepsjeng, bc-twitter, pr-twitter 등에서 marginal loss (max -11.8%).
- PACT는 P50에서 Colloid/NBT/Memtis 대비 각각 **9.7%, 5.0%, 16.6% 평균 향상** (1:2). Peak improvement **61%** (2:1).

### vs. Soar (Offline Profiling, Figure 6)

Soar의 offline profiling 대비 PACT의 online-only 접근 평균 **3.3% 추가 slowdown**만 발생. Cost-free online adaptation의 feasibility 입증.

### PAC vs. Frequency 내부 비교 (Figure 9)

동일 promotion count에서 PAC 기반 selection이 frequency 기반 selection보다 **18%** 성능 향상. PAC는 execution 초기에 promotions 집중, frequency는 oscillatory burst 패턴. MLP variance가 큰 workload에서 최대 22% 차이.

### Bandwidth Contention (Figure 11)

MLC로 DRAM bandwidth saturation (1-8 threads). 모든 contention level에서 PACT가 Colloid(4KB)/Memtis(THP)와 동등 이상 성능 + **3.5-4.7× 적은 promotions**.

### Colocation (Figure 12)

Sequential(high-MLP) + Random(low-MLP) Masim processes colocation: PACT가 Colloid 대비 sequential 112%, random 28%, aggregate **61%** slowdown 감소. Uniform attribution이 mixed access pattern에서도 robust.

### Sensitivity Analysis (Figure 10)

| Parameter | Range | Impact |
|-----------|-------|--------|
| PEBS rate | 800 → 4000 | Slowdown 23% → 30% 선형 증가 |
| Sampling period | 10ms → 1000ms | 20ms optimal, 1000ms에서 slowdown 20% → 27% |
| Cooling factor | Base/Reset/Decay-2 | Cooling 사용 시 성능 동등 또는 저하 (PACT는 이미 충분히 responsive) |

Default parameters (PEBS 400, 20ms, α=1.0)가 모든 workload에서 workload-specific optimum의 5% 이내.

## 구현

- **Platform:** Linux 5.15, TPP의 일부 module 통합 (LRU demotion reliability fix)
- **Custom optimizations:**
  - PEBS kernel interface: 불필요 field 제거 → 5MB compact buffer.
  - Linux perf subsystem 확장: PMU counter로부터 PAC direct computation.
  - Shared-memory buffer로 perf↔PACT low-overhead communication.
- **Threads:** 2개 dedicated (PEBS processing, migration)
- **Memory overhead:** 25 bytes/page (0.6%)
- **NUMA hint fault scanning disabled** (PACT 자체 policy와의 충돌 방지)
- **Open-source:** `https://github.com/MoatLab/PACT`
