# CXLMC: Model Checking CXL Shared Memory Programs

**Venue:** ASPLOS '26 (Vol. 2), March 22–26, 2026, Pittsburgh, PA, USA
**저자:** Simon Guo, Conan Truong, Brian Demsky

## 개요

CXL shared memory는 최대 4,095개의 memory entity가 cache-coherent하게 단일 memory pool을 공유하게 해주지만, cluster 규모가 커질수록 **machine failure**가 불가피해짐. 특히 compute node failure가 주요 위험 요소.

**CXL failure의 특수성:**
- CXL은 latency hiding을 위해 remote memory를 CPU cache에 캐싱. Machine failure 시 dirty cache line이 flush되지 않으면 **최신 store가 유실**됨.
- CXL spec은 **Global Persistent Flush (GPF)** 메커니즘을 정의하나, GPF는 power outage 시나리오에 초점을 두고 있으며 battery/capacitor 기반. Motherboard, CPU, CXL transceiver, cable failure 등에서는 GPF가 작동하지 않을 수 있음.
- CXL의 coherence directory 구조는 어떤 cache line이 lost되었는지 알 수 있게 함 → **memory poisoning** 가능 (opt-in). 단, 현재 poisoning을 활용하도록 설계된 application은 없음.

**PM 대비 CXL 차이점:**
1. **Partial failure model**: PM은 total failure (모든 thread crash + quiescent period). CXL은 개별 machine failure 동안 다른 machine은 계속 실행.
2. **Coherence로 인한 implicit write-back**: Remote machine이 cache line을 load하면 coherence protocol에 의해 해당 line이 CXL memory device로 write-back됨.
3. **Concurrent failure + recovery**: Recovery 도중에도 추가 failure 발생 가능.

**Crash consistency의 어려움:**
- PM 분야에서 183개의 crash-consistency bug가 tool들에 의해 발견됨 (적은 수의 benchmark에서). Tool 없이는 bug-prone.
- CXL shared memory는 아직 hardware가 상용화되지 않았으나, 이미 database indices [36, 39, 40, 51], message buffers [8, 56, 57], shared object stores [63, 72] 등 **핵심 shared data structure**들이 CXL로 구현되고 있음. 단일 machine crash가 cluster 전체를 멈출 수 있음.

## 방법론

### 1. System Overview (§3)

CXLMC는 constraint-based model checking을 통해 CXL shared memory program의 crash-consistency bug를 탐지. LLVM compiler pass로 instrumentation 후, multi-process로 실행하며 CXL shared memory를 emulate.

**System configuration (§3.1):**
- Multiple x86 compute nodes → single CXL memory device
- 각 process가 하나의 machine을 simulate, process 내 multiple threads 지원
- Memory model: x86-TSO + flush instructions (clflush, clflushopt, clwb)
- Memory poisoning 지원 (optional, 평가에서는 미사용)

**Correctness guarantees (§3.2):**
- **Soundness**: CXLMC가 찾은 모든 execution은 memory/failure model 하에서 feasible → 모든 bug는 actual bug.
- **Completeness**: Fixed schedule에 대해 reads-from equivalence class별 최소 1개 execution enumerate. Concurrency non-determinism은 model check하지 않고 random seed로 schedule 다양화 → longer execution으로 scaling.
- **Scope**: Assertion failure / crash 탐지. Robustness 등 추가 property check로 확장 가능하도록 설계.

### 2. Constraint Refinement for Partial Failures (§3.3)

**Naive enumeration 문제:**
- Yat [47]의 eager enumeration은 PM에서도 intractable. CXL에서는 매 step마다 모든 subset of processes가 fail 가능 → 훨씬 더 심각한 state explosion.
- 해결: Jaaru [33]의 **cache line constraint refinement**를 CXL partial failure에 adapt.

**Cache line constraint 개념:**
- Post-failure state는 failure 전 마지막 write-back 시간에 의해 결정. 그 시간 이전의 store들은 persisted, 이후 store들은 lost.
- 각 cache line에 대해 **constraint interval [begin, end)** 유지. 이 interval은 가능한 write-back timestamp 범위를 encoding.
- Constraint refinement는 commit store pattern마다 2개의 execution만 explore: failure-before-commit-store-writeback vs no-failure.

**CXL adaptation — per-machine constraints:**
- PM에서는 single failure domain → 단일 constraint set. CXL에서는 각 failed machine μ마다 separate constraint (μ.cacheline(addr)) 유지 (Figure 3, 4).
- 예: Machine A가 fail하고 B가 x를 load → A의 constraint 사용. B가 y를 load → B의 constraint 사용. 최대 k개의 constraint per cache line (k = failed machines 수).

**Constraint update triggers:**
- **clflush addr** → lower bound를 현재 sequence number(σcurr)로 set (Commit_SB)
- **Remote load from addr (CXL coherence)** → load한 machine의 cache로 인해 write-back 발생 → constraint lower bound update (Commit_FB, Algorithm 3)
- **Post-failure load result** → 읽은 값에 따라 constraint interval refine (Algorithm 4, Figure 2 예시: x=2 읽으면 [2,4), y=3 읽으면 [4,7))

### 3. Memory Model Implementation (§4.1)

CXLMC는 Jaaru의 store buffer 기반 emulation을 채택, CXL 특성에 맞게 수정:

**Per-thread state:**
- `Sτ` (store buffer): store, sfence, clflush instructions queue (FIFO commit)
- `Fτ` (flush buffer): clflushopt instructions queue (weaker ordering)
- `queue(addr)`: globally visible stores → ⟨val, σ, μ⟩ triplets (sequence number + issuing machine 포함)

**x86-TSO ordering (Table 1):**
- Store → later store: preserved
- Store → later load (same thread): preserved (bypass)
- clflushopt ↔ stores (different cache line): may reorder
- mfence: empties Sτ+Fτ, serializes all
- sfence: empties Fτ only

**CXL-specific modification:**
- `μ.cacheline(addr)`는 thread epoch 대신 **per-machine**으로 관리
- Commit_FB(addr, σ): flush buffer의 clflushopt가 cache line constraint의 lower bound를 max(cl.begin, σ)로 update
- Remote load 시 commit store pattern: 다른 machine에서 load하면 해당 cache line이 write-back됨 → constraint update

## 핵심 기여

CXLMC는 **CXL shared memory의 crash consistency를 model checking하는 최초의 tool**. 핵심 contribution:

1. **Constraint-based model checking의 CXL adaptation**: PM의 total failure → CXL의 partial failure로 전환. Per-machine constraint, concurrent failure + recovery, CXL coherence의 implicit write-back 특성 반영.
2. **Multi-process model checking**: Multiple Linux process로 CXL cluster emulation → 최초의 multi-process crash-consistency model checker.
3. **실질적 bug 발견**: 8개 application에서 24개 bug 발견 (7개 new). CXL shared memory가 실용화되기 전에 software stack의 crash-consistency robustness 확보에 기여.

**Broader significance:** CXL shared memory가 상용화되면 cluster-wide coordination data structure의 crash consistency가 critical해질 것. CXLMC는 아직 hardware가 없는 상황에서 software-level correctness를 선제적으로 검증할 수 있는 infrastructure를 제공. Memory poisoning, robustness 등으로 확장 가능한 설계.

## 주요 결과

**ScanStores(addr, Φ, σ_start) (Algorithm 3):** 가능한 post-crash store values enumeration. 조건:
- Store sequence number ≤ σ_start
- Φ에 속한 machine의 store는 해당 machine의 cacheline constraint 이후에 발생한 것만 visible (이전 것들은 write-back으로 persisted)
- Failed machine의 store는 해당 machine의 cacheline constraint end보다 작아야 함

**BuildMayReadFrom(addr) (Algorithm 3 Line 7–10):** Load가 읽을 수 있는 값 결정. Store buffer에 matching addr의 store가 있고 이후에 같은 addr의 store가 없으면 → local bypass로 ⟨μ_curr, 0, val, Φ_curr⟩ 반환. 그렇지 않으면 ScanStores로 global queue 조회.

**Explore(execution e) (Algorithm 5):**
- Non-deterministically choose: commit store buffer head OR execute next action
- Load 시 BuildMayReadFrom으로 가능한 모든 value에 대해 분기 (DoRead)
- Flush instructions that increment cache line constraint + past a store from μ∉Φ_curr → Fail(μ) 분기 (해당 machine failure simulate)
- Fail(μ): μ를 Φ_curr에 추가, 해당 machine의 thread stop
